rtifact Registry

## Purpose

This document indexes important repository artifacts as part of PAM’s file-first observatory model.

It is **not** yet a complete registry of every output in the repository. Its purpose is narrower:

- make the current artifact spine inspectable
- clarify which layers are canonical runtime outputs versus downstream canonicalization outputs versus repository-facing observatory artifacts
- record scope notes and caveats that matter for scientific trust
- help future validation and audit work attach claims to concrete files

This registry should grow over time.

---

## Reading note

PAM is a **file-first observatory**.

That means many important scientific objects exist as explicit artifacts rather than only as in-memory runtime state. The canonical runtime, downstream canonicalization scripts, and repository-facing observatory layers all depend on inspectable files.

A reader should therefore not ask only:
- what code exists?

but also:
- what artifacts exist?
- who produced them?
- what inputs do they assume?
- what status do they currently have?
- what caveats or scope limits must be preserved?

---

## Status vocabulary

### Layer
Broad observatory layer to which the artifact belongs.

Typical values:
- `pipeline`
- `corpus`
- `measurement`
- `observables`
- `geometry`
- `phase`
- `operators`
- `initial_conditions`
- `topology`
- `family_gateway`
- `response_flow`
- `attractor_basin`
- `linked_response_taxonomy`

### Status
Current maturity of the artifact.

Typical values:
- `canonical_runtime`
- `downstream_canonical`
- `repository_facing`
- `provisional_study`

Status describes the artifact’s current repository role, not a claim that the underlying scientific interpretation is final.

### Scope / caveat
Short note preserving:
- file-first assumptions
- model-specific limits
- subset-specific limits
- provenance limitations
- not-yet-promoted architectural status

## Pipeline scope note

The PAM Observatory is a **file-first** system.

The canonical runtime entrypoint:

- `scripts/run_full_pipeline.sh`

currently orchestrates **derived-stage analysis over an existing artifact store**.

In its present form, the canonical runtime rebuilds:

- geometry
- phase
- operators
- initial conditions
- topology

from existing upstream outputs.

It should **not** be read as a single raw-corpus-to-final-result pipeline. Upstream corpus generation, measurement, and observable construction are part of the broader instrument, but they are not all regenerated by this entrypoint.

This is intentional. The file-first design makes intermediate layers inspectable, reusable, and auditable.

## Current priorities for expansion

The next useful additions to this registry are likely:

1. more explicit upstream corpus-generation artifacts
2. clearer measurement / observable-layer artifact rows
3. direct links from artifacts to validators where available
4. more complete response-flow artifact coverage
5. more explicit identity / holonomy / obstruction artifact surfaces
6. versioning or date fields where artifact maturity becomes sensitive to run provenance

---

## Guidance for future updates

When adding a new artifact row, try to include:

- the file or directory name as it appears in the repo
- the script or module that most directly produces it
- the main input layer it depends on
- whether it is:
  - canonical runtime
  - downstream canonical
  - repository-facing
  - provisional study
- one honest caveat or scope condition
- a doc or OBS note that explains how to read it

This registry is most useful when it preserves limitations as well as outputs.

---

## Bottom line

The registry exists to make the file-first observatory model inspectable.

It should help a reader answer:

- what is the artifact?
- where did it come from?
- what role does it play?
- how mature is it?
- what should I be careful not to overclaim from it?
