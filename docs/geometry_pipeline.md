# Geometry Pipeline
### PAM → Fisher Geometry → Distance → Manifold

This document explains the **geometry analysis pipeline** used in the PAM Observatory.

The goal of the pipeline is to translate **raw experiment dynamics** into a **geometric description of the parameter manifold**.

Instead of analyzing individual trajectories in isolation, the system measures how observable statistics change across parameter space and then derives the **information geometry** induced by those changes.

---

# Conceptual Pipeline

The repository implements the following layered workflow:

```
quench experiments
        ↓
trajectory statistics
        ↓
observable phase surfaces
        ↓
Fisher-type metric tensor
        ↓
Fisher geodesic distances
        ↓
manifold embedding (MDS)
        ↓
phase geometry interpretation
```

Each stage transforms the representation of the system while preserving the underlying dynamics.

---

# 1. Quench Experiments

Experiments are generated by sweeping the parameter manifold

```
θ = (r, α)
```

across a grid.

Typical configuration:

```
r ∈ {0.10, 0.15, 0.20, 0.25, 0.30}
α ∈ linspace(0.03, 0.15, 15)
seeds = 10
```

Total runs:

```
5 × 15 × 10 = 750 quenches
```

Each run produces:

* trajectory snapshots
* entropy measurements
* freeze dynamics
* correlation statistics

and writes a summary row to:

```
outputs/index.csv
```

This file acts as the **canonical experiment ledger**.

---

# 2. Observable Surfaces

Trajectory statistics are aggregated into observable summaries.

Examples:

```
piF_tail
H_joint_mean
corr0
best_corr
delta_r2_freeze
delta_r2_entropy
K_max
```

These values form a vector:

```
m(r, α)
```

which describes the macroscopic behavior of the system at each parameter point.

When visualized across the grid, these values produce **phase surfaces**.

---

# 3. Fisher-Type Metric

The next step converts observable gradients into a local geometric metric.

The Fisher-type metric is defined as

```
g_ij = ∂_i mᵀ Σ⁻¹ ∂_j m
```

where

* `m(r, α)` is the observable vector
* `Σ` is the empirical covariance of observable noise across seeds

This metric measures how **distinguishable two nearby parameter points are** in terms of observable behavior.

Large values indicate strong sensitivity to parameter changes.

Outputs include:

```
fim_rr
fim_ra
fim_aa
det(g)
eigvals
condition_number
```

These are written to:

```
outputs/fim/
```

---

# 4. Fisher Distance

The metric defines local geometry, but not global distances.

To recover the intrinsic geometry of the manifold we construct a **Fisher-distance graph** across the parameter grid.

Edges connect neighboring parameter points with weights derived from the metric.

From this graph we compute:

```
geodesic distances
shortest paths
distance matrices
```

These are stored in:

```
outputs/fim_distance/
```

---

# 5. Manifold Embedding (MDS)

Once geodesic distances are known, the manifold can be embedded in a lower-dimensional space using **Multidimensional Scaling (MDS)**.

This produces coordinates:

```
mds_coords.csv
```

and visualizations such as:

```
mds_embedding.png
mds_embedding_colored_by_det.png
mds_embedding_colored_by_phase.png
```

These embeddings reveal the **shape of the parameter manifold**.

In many systems, distinct phase regimes appear as **separated lobes or folds** of the embedding.

---

# 6. Curvature and Phase Boundaries

Curvature diagnostics are applied to detect candidate phase boundaries.

Indicators include:

* metric determinant ridges
* eigenvalue anisotropy
* condition number spikes

These diagnostics are written to:

```
outputs/fim_curvature/
```

Regions where curvature becomes large often correspond to **phase transitions** in the underlying dynamics.

---

# The Big Picture

The geometry pipeline transforms raw dynamics into a geometric object.

```
language dynamics
        ↓
statistical fingerprints
        ↓
information geometry
        ↓
phase manifold
```

The PAM Observatory exists to measure and visualize this structure.

# Design Principles

The pipeline follows several guiding rules.

### Layer separation

Each stage of the analysis is implemented in a separate module:

```
dynamics          → exp_batch.py
statistics        → common_quench_metrics.py
phase surfaces    → plot_phase_surfaces.py
metric estimation → fim.py
distances         → fim_distance.py
embedding         → fim_mds.py
curvature         → fim_curvature.py
```

### Observation-driven state

All geometry computations derive their state from observable artifacts such as:

```
outputs/index.csv
trajectory files
fim surfaces
distance matrices
```

This avoids hidden state and ensures reproducibility.

### Reconciliation over assertion

System state should be inferred from what **exists on disk**, not from assumptions made during runtime.

---

# Why Geometry?

Many complex systems exhibit phase structure that is difficult to see directly in trajectory data.

Information geometry provides a way to reveal this structure by measuring **how system behavior changes across parameter space**.

The result is a geometric object — a manifold — whose curvature, folds, and boundaries correspond to qualitative changes in system behavior.

---

# The Observatory View

Within the repository, the full stack becomes:

```
experiments → observables → geometry → interpretation
```

The observatory infrastructure exists to make each of these layers visible while experiments are still unfolding.

In this sense, the repository is not just a collection of scripts — it is an **instrument for exploring the geometry of recursive language systems**.
